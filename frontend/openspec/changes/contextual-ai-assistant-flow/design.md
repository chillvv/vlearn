## Context

当前前端以 `/chat` 作为主要 AI 入口，录入、解析、修订和学习复盘在不同页面割裂，且上下文无法自动继承。节点详情页虽然存在 AI 模块，但是静态示例，不支持结构化数据驱动与局部追问。此次改造需要跨越路由层、页面交互层、数据契约层与展示组件层，并且要在 MVP 阶段控制复杂度。

## Goals / Non-Goals

**Goals:**
- 将 AI 从“全局聊天入口”迁移为“场景化助手”：录入确认页 + 详情页抽屉。
- 建立结构化 JSON 驱动的详情展示，字段为空即折叠，不出现空白模块。
- 增加 `original_image_url`，先做图像增强与原图保真，不做生成式重绘。
- 详情页支持步骤化解析与公式渲染能力，兼容 Markdown 表格。
- 保留轻量用户干预：报告错误、个人笔记置顶。

**Non-Goals:**
- 不实现全局悬浮球聊天。
- 不在首期实现复杂规则引擎（题型规则树）。
- 不在首期实现多风格输出和 AI 重写多版本管理。
- 不实现复杂图形（几何、电路）重建绘图。

## Decisions

1. **入口迁移：移除 `/chat`，改为流程内触发**
   - 选择：删除侧边栏 AI 聊天入口与路由页面入口。
   - 原因：避免脱离题目上下文，减少学习路径跳转成本。
   - 备选：保留聊天页并弱化入口；被拒绝，因为仍会分散注意力。

2. **录入流程：新增“AI 草稿确认页”作为保存前闸门**
   - 选择：录入后先展示结构化草稿，用户确认后入库。
   - 原因：避免“静默保存”造成误分类与后续复盘污染。
   - 备选：继续直接入库 + 事后编辑；被拒绝，因为纠错链路更长。

3. **数据契约：结构化 JSON + 空值折叠**
   - 选择：以 `core_reason`、`detailed_steps[]`、`formula_or_rule`、`prerequisite_knowledge`、`warning_tags`、`original_image_url` 为主契约。
   - 原因：让前端渲染稳定可控，避免纯文本盲盒。
   - 备选：纯 Markdown 全量渲染；被拒绝，因为结构与可见性不可控。

4. **理科展示：LaTeX 与 Markdown 双渲染**
   - 选择：公式字段与步骤内容支持 LaTeX；前置知识支持 GFM 表格。
   - 原因：覆盖数学/物理/化学核心表达，且开发成本可控。
   - 备选：自研复杂表格结构与图形 DSL；被拒绝，因为超出 MVP。

5. **AI 交互容器：详情页 Drawer / Bottom Sheet**
   - 选择：PC 右侧抽屉，移动端 Bottom Sheet，不跳转。
   - 原因：保留主页面上下文，符合学习流连续性。
   - 备选：全屏聊天页；被拒绝，因为会中断复盘流程。

## Risks / Trade-offs

- [模型输出不稳定] → 通过 Schema 校验 + 空值回退 + 最小可用模板兜底。
- [Markdown/LaTeX 渲染安全与样式风险] → 使用受控渲染链路和样式白名单，禁止危险 HTML。
- [移动端抽屉交互复杂] → 统一复用现有 Drawer/Sheet 组件并增加断点行为测试。
- [录入流程变长影响转化] → 默认提供“准确无误，存入错题库”主按钮，减少确认页操作负担。

## Migration Plan

1. 新增草稿确认页与结构化卡片组件，不影响现有错题库读取接口。
2. 在路由与侧边栏移除 `/chat` 入口，同时把“召唤 AI”能力下沉到详情页局部触发器。
3. 节点详情页按结构化字段改造展示模块，并接入公式/Markdown 渲染。
4. 接入 `original_image_url` 展示与图像增强占位接口（MVP 可先前端保留字段）。
5. 完成构建与回归验证后发布；如异常可回滚到旧详情展示组件与旧路由配置。

## Open Questions

- 图像增强服务首期由前端本地处理还是后端统一接口处理。
- 结构化草稿是否需要“快捷修订指令胶囊”首期一并上线。
- `detailed_steps` 缺失时，是否自动将 `note` 退化为单步解析展示。
